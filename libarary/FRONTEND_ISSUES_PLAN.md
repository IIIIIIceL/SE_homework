# 前端开发 Issue 规划

## 📋 概述

本文档定义了图书馆管理系统前端开发的全部 Issue，分为 8 个阶段，由熊昱浩和谷雨萌两人轮流负责。  
每个 Issue 都包含具体的子任务和验收标准，遵循 Git 规范管理。

---

## 🔧 Git 规范约定

### 分支命名
- 格式：`feat/fe-{序号}-{短描述}` 
- 示例：`feat/fe-01-bootstrap`、`feat/fe-04-books`

### Commit 规范
- 格式：`[FE-{序号}] {动作}: {描述}`
- 示例：`[FE-01] feat: add http client and request interceptor`
- 支持的动作：`feat`、`fix`、`refactor`、`docs`、`test`

### Pull Request
- 标题：`[FE-{序号}] {功能名}`
- 正文：关键改动点、关联 Issue 编号、测试方式
- 至少需要另一人 review

### FE-02 登录态与权限骨架

**负责人：** 谷雨萌  
**分支：** `feat/fe-02-auth-guard`  
**优先级：** ⭐⭐⭐⭐⭐ 最高（前置依赖）

#### 📝 背景
用户身份认证是系统的基础。需要实现登录流程、token 管理、路由守卫，确保未授权用户无法访问受限页面。

#### 🎯 范围
- 实现登录页面和登录逻辑
- Token 存储与读取机制
- 路由守卫与重定向
- 登出功能
- 401/403 统一处理

#### ✅ 子任务
- [ ] 创建登录页面 (`src/pages/Login/index.jsx`)
  - 用户名/密码表单
  - 错误提示
  - 登录按钮（禁用、加载态）
  - 链接到注册页（可选，后续扩展）

- [ ] 实现登录服务 (`src/services/authService.js`)
  - `login(username, password)` 方法
  - 调用后端 `/api/auth/login` 接口
  - 返回 token 和用户信息

- [ ] 实现 Token 存储管理 (`src/utils/tokenManager.js`)
  - `setToken(token)` - 存储到 localStorage
  - `getToken()` - 获取 token
  - `clearToken()` - 清除 token
  - `isTokenValid()` - 检查 token 有效性（可选）

- [ ] 创建全局认证上下文 (`src/context/AuthContext.jsx`)
  - 管理当前登录用户信息
  - 提供 login、logout、checkAuth 方法
  - 支持页面刷新时恢复登录态

- [ ] 完善路由守卫 (`src/components/ProtectedRoute.jsx`)
  - 检查 token 是否存在
  - token 不存在时重定向到登录页
  - 页面刷新时重新验证 token（调用后端验证接口）

- [ ] 实现登出功能
  - 清除本地 token
  - 清除用户信息
  - 重定向回登录页

- [ ] 处理 401/403 错误
  - HTTP 拦截器捕获 401 响应
  - 自动清除 token 并重定向到登录页
  - 显示友好的错误提示

- [ ] 创建 `src/pages/Unauthorized.jsx` 页面
  - 展示 403/权限不足提示
  - 返回首页链接

#### 🏆 验收标准
- ✓ 登录页面能正常显示，表单验证完整
- ✓ 正确的用户名密码可以登录成功
- ✓ Token 正确存储到 localStorage
- ✓ 登出后 token 被清除
- ✓ 没有 token 访问受限页面时，自动重定向到登录页
- ✓ 页面刷新后，如果有有效 token，能恢复登录态
- ✓ 后端返回 401 时，自动处理并重定向
- ✓ HTTP 拦截器能在请求头中自动加入 Authorization: Bearer {token}
